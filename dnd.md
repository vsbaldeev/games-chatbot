# D&D-приключение — техническая документация

D&D-приключение — многопользовательская игра для 3+ участников. Бот генерирует абсурдный фэнтезийный сценарий, игроки выбирают действия, бот бросает d20 за каждого и просит LLM рассказать, что произошло. Игра длится 1, 2 или 3 раунда в зависимости от команды запуска.

---

## Запуск

### Команды

| Команда | Раундов | Режим |
|---------|---------|-------|
| `/dnd_pvp`  | 1 | PvP — все против всех |
| `/dnd_coop` | 2 | Кооп — отряд vs. Босс |
| `/dnd_heist`     | 3 | Великое Ограбление (стелс и хитрость) |

Доступны только в группах (`ChatType.GROUPS`). Вызвавший автоматически становится первым участником.

В чате может идти только одна игра одновременно. Если игра уже активна — бот сообщает об этом.

---

## Фаза 1: лобби

Бот отправляет одно сообщение с двумя кнопками. Присоединиться может любой участник чата.

```
⚔️ D&D Приключение (3 раунда)

Набирается отряд для опасного похода!
Нужно минимум 3 игрока.

Отряд (2) — нужно ещё 1:
• @player1
• @player2

Лобби закроется через 5 минут.

[⚔️ Присоединиться]  [▶️ Начать (2/3)]
```

После набора 3+ игроков кнопка Start меняет подпись на `▶️ Начать (N игроков)`.

Состояние хранится в:
```
__lobbies:             dict[chat_id → (LobbyState, max_rounds)]
__lobby_timeout_jobs:  dict[chat_id → Job]  # таймер 300 сек
```

### Защита от чужих нажатий

| Действие | Результат |
|----------|-----------|
| Нажать «Присоединиться» повторно | Алерт «Ты уже в отряде!» |
| Нажать «Начать» с < 3 игроков | Алерт «Нужно ещё N игрока для старта.» |

Начать может любой — ограничений по роли нет.

### Таймаут лобби (5 минут)

Если никто не нажал «Начать», лобби закрывается: сообщение редактируется, кнопки убираются, чат освобождается.

## Фаза 2: генерация сценария

После нажатия «Начать» лобби уничтожается и сообщение редактируется:

```
⚔️ D&D Приключение

🎲 Генерация сценария...
```

Запускается job (`__start_game_job`), который вызывает LLM для генерации первого раунда.

### LLM: генерация раунда

Модель: `llama-3.3-70b-versatile` (Groq).

**Системный промпт** задаёт строгий формат ответа:

```
СЦЕНАРИЙ: <одно-два предложения>
Д1: <действие 2-5 слов>
Д2: <действие 2-5 слов>
Д3: <действие 2-5 слов>
Д4: <действие 2-5 слов>
```

**Пользовательский промпт** для раунда 1 просит придумать абсурдный фэнтезийный сценарий. Для раунда 2+ в промпт включается история предыдущих раундов (сценарий + нарратив), чтобы продолжение было связным. Финальный раунд помечается отдельной инструкцией — LLM должен придумать кульминацию.

**Парсинг:** `__parse_scenario` разбирает ответ построчно. Если сценарий или часть действий не распознана — подставляются заглушки (`"Бежать"`, `"Атаковать"` и т.д.).

### Режим ограбления (`/dnd_heist`)

Три фазы, каждая — отдельный раунд:

| Раунд | Название | Задача |
|-------|----------|--------|
| 1 | Проникновение | Попасть внутрь незаметно |
| 2 | Дело | Добраться до цели и взять трофей |
| 3 | Побег | Уйти от погони |

Действия — только стелс и хитрость (отвлечь охрану, взломать магический замок, притвориться официантами и т.п.). Лобовые атаки в промпт не включаются. Нарратив оценивает успех каждого игрока по броску d20 в контексте выбранного трюка.

### Кооп-режим (`/dnd_coop`)

Формат LLM в раунде 1 включает дополнительное поле `БОСС:`:

```
СЦЕНАРИЙ: <одно-два предложения — встреча с боссом>
БОСС: <смешное имя босса (3-6 слов)>
Д1: <кооперативное действие 2-5 слов>
...
```

Имя босса фиксируется на всю игру. В раунде 2 LLM генерирует продолжение битвы (без поля БОСС).

**HP босса** задаётся при старте: `random.randint(N * 15, N * 20)`, где N — число игроков. Для трёх игроков это 45–60 HP. Средний суммарный урон за 2 раунда: `3 * 10.5 * 2 ≈ 63`, что даёт ~70% вероятность победы.

**Разрешение кооп-раунда:** броски всех игроков суммируются как урон боссу. Победа — если `boss_hp ≤ 0` после финального раунда.

## Фаза 3: голосование

Сообщение редактируется, показывается сценарий и 4 кнопки действий в сетке 2×2. У каждого игрока **45 секунд** (`DND_ACTION_TIMEOUT`).

```
⚔️ D&D — Раунд 2/3

Пьяный гоблин-брокер предлагает вложиться в «очень надёжный» вулкан.

У каждого игрока 45 секунд на выбор:
✅ @player1
⏳ @player2
⏳ @player3

[Вложить всё]        [Обыскать гоблина]
[Столкнуть в вулкан] [Попросить проспект]
```

По мере нажатий сообщение редактируется: `⏳` меняется на `✅`.

Состояние хранится в:
```
__active_games:       dict[chat_id → ActiveGame]
__action_timeout_jobs: dict[chat_id → Job]  # таймер 45 сек
```

### Защита от чужих нажатий

| Кто нажал | Результат |
|-----------|-----------|
| Не участник игры | Алерт «Ты не в этой партии, зритель! 👀» |
| Участник, уже выбравший действие | Алерт «Ты уже выбрал: <действие>» |

### Завершение голосования

Если все игроки выбрали действие до истечения таймера — голосование завершается досрочно, таймер отменяется. Если таймер истёк — голосование завершается принудительно; игроки, не успевшие выбрать, получают действие `"бездействовал"`.

## Фаза 4: разрешение раунда

Запускается `__resolve_game_job`. Сообщение редактируется:

```
⚔️ D&D — Раунд 2/3

🎲 Подводим итоги...
```

Затем:

1. Бот бросает `random.randint(1, 20)` за каждого игрока.
2. LLM получает сценарий, список игроков с действиями и бросками.
3. LLM возвращает единый связный нарратив.

### LLM: генерация нарратива

В промпт передаётся:
- История предыдущих раундов (если есть)
- Текущий сценарий
- Список: `@username выбрал "действие" → 🎲roll (пометка)`

Бросок 1 помечается `(КРИТИЧЕСКИЙ ПРОВАЛ)`, бросок 20 — `(КРИТИЧЕСКИЙ УСПЕХ)`. LLM получает явную инструкцию описать соответствующие исходы как нечто особенно абсурдное.

Для финального раунда инструкция меняется: завершить всю историю эффектно и окончательно.

### Сообщение с итогом раунда

Кнопки убираются, сообщение редактируется:

```
⚔️ D&D — Раунд 2/3 — Итог

@player1 вложил все деньги и выпало 20 — монеты превратились
в золотых драконов, которые сожрали гоблина...

🎲 Броски:
• @player1: Вложить всё → 🎲20 ✨
• @player2: Обыскать гоблина → 🎲7
• @player3: бездействовал → 🎲3 💀
```

Финальный раунд:

```
⚔️ D&D — Финал

И вот, после трёх раундов абсурда, отряд наконец...

🎲 Броски:
• ...
```

## Переход между раундами

Если это не финальный раунд (`game.round_number < game.max_rounds`):

1. Нарратив и результаты сохраняются в `game.history`.
2. `game.round_number` увеличивается на 1.
3. `game.choices` сбрасывается.
4. Запускается `__next_round_job`: отправляется **новое сообщение** (не редактирование) с пометкой «Генерация продолжения...».
5. LLM генерирует следующий сценарий с учётом истории.
6. Новое сообщение редактируется — показываются кнопки действий.
7. Запускается новый таймер на 45 секунд.

Каждый раунд — отдельное сообщение в чате, результаты накапливаются и видны в истории переписки.

## Структуры данных

```python
@dataclass
class LobbyState:
    chat_id: int
    message_id: int
    initiator_id: int
    players: list[tuple[int, str]]   # (user_id, username)

@dataclass
class ActiveGame:
    chat_id: int
    message_id: int        # id текущего активного сообщения
    scenario: str
    actions: list[str]     # 4 действия текущего раунда
    players: list[tuple[int, str]]
    max_rounds: int        # задаётся командой запуска (1/2/3)
    round_number: int      # текущий раунд, начиная с 1
    mode: str              # "pvp" | "coop" | "adventure"
    boss_name: str         # только для coop
    boss_hp: int           # текущие HP босса (только для coop)
    boss_max_hp: int       # максимальные HP босса (только для coop)
    choices: dict[int, int]  # user_id → индекс действия
    history: list[dict]    # [{scenario, narrative, results}, ...]
```
